# ПрофДНК: минимальная версия (ТИТАНИТ 2026)

Рабочая минимальная версия по кейсу хакатона: онлайн-платформа для профориентологов с конструктором методик, прохождением по ссылке без регистрации, сохранением результатов и генерацией двух типов отчетов (HTML + DOCX).

## Что уже реализовано

### Роли
- `admin`: создает психологов, управляет доступом и блокировкой.
- `psychologist`: создает тесты, работает с результатами и отчетами.
- `client`: проходит тест без регистрации по персональной ссылке.

### Сквозной сценарий
1. Психолог создает тест (через UI или импорт JSON).
2. Психолог отправляет клиенту ссылку.
3. Клиент проходит тест и отправляет ответы.
4. Система считает метрики и сохраняет прохождение.
5. Психолог получает клиентский и профессиональный отчеты.

### Покрытие кейса
- Конструктор с секциями и типами вопросов:
  - `text`, `textarea`, `single_choice`, `multiple_choice`, `yes_no`, `number`, `slider`, `datetime`, `rating`.
- В конструкторе доступны 3 быстрых пресета методик под разные аудитории.
- Редактор формул производных метрик.
- Импорт/экспорт теста в JSON.
- Клонирование теста в один клик (копия структуры, полей клиента, формул и шаблонов отчётов).
- Публичная визитка психолога + QR-код.
- Отчеты генерируются по запросу и не хранятся отдельными файлами на диске.
- Поддержка именованных ссылок кампаний и группировка результатов по источникам.
- Сравнение кампаний по среднему % балла и % заполнения прямо на странице теста.
- CSV-экспорт всех прохождений теста в один клик.
- Rate limit для защиты от перебора на `POST /login` и от спама на `POST /t/{token}`.

## Технологии
- Бэкенд: `Python 3.11`, `FastAPI`, `SQLAlchemy`.
- Миграции: `Alembic`.
- База данных: `PostgreSQL`.
- Интерфейс: `Jinja2 + vanilla JS`.
- Отчеты: `python-docx` + HTML-шаблоны.
- Развертывание: `Docker Compose`, `systemd`, `nginx`.

## Локальный запуск (без Docker)

1. Установить зависимости:
```bash
pip install -r backend/requirements.txt
```

2. Настроить переменные окружения:
- заполнить `.env` (или экспортировать переменные вручную);
- минимально нужны:
  - `APP_SECRET_KEY`
  - `DATABASE_URL`
  - `BASE_URL`
- рекомендовано:
  - `APP_ENV=production` для прод-сервера;
  - `SESSION_HTTPS_ONLY=true` на HTTPS-домене;
  - `SESSION_SAME_SITE=lax` (или `strict` при необходимости).
- для управления схемой БД:
  - `AUTO_CREATE_SCHEMA=false` в production (схема только через Alembic);
  - `AUTO_CREATE_SCHEMA=true` в локальной разработке (быстрый старт).
- для логирования:
  - `LOG_LEVEL=INFO` (или `DEBUG` для детальной диагностики).
- для защиты от частых запросов:
  - `LOGIN_RATE_LIMIT_ATTEMPTS=8`
  - `LOGIN_RATE_LIMIT_WINDOW_SECONDS=300`
  - `SUBMIT_RATE_LIMIT_ATTEMPTS=30`
  - `SUBMIT_RATE_LIMIT_WINDOW_SECONDS=60`
- для почты (опционально):
  - `SMTP_ENABLED=true`
  - `SMTP_HOST=smtp.gmail.com`
  - `SMTP_PORT=587`
  - `SMTP_USER=...`
  - `SMTP_PASSWORD=...`
  - `SMTP_FROM=...`
  - `SMTP_TLS=true`
- `AUTO_SEED` по умолчанию выключен (`false`) и включается явно для демо-стендов.

3. Применить миграции:
```bash
cd backend
alembic upgrade head
```

4. Запустить приложение:
```bash
cd backend
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

## Автодеплой на VPS (GitHub Actions)
- Воркфлоу: `.github/workflows/deploy.yml`
- События запуска: `push` в `main` и ручной запуск `workflow_dispatch`.
- Целевой сервер: `recallio.tech`.
- На сервере выполняется:
  - обновление репозитория в `/srv/titanit-profdnk-mvp`,
  - обновление зависимостей Python,
  - перезапись `.env.deploy`,
  - применение миграций `alembic upgrade head`,
  - `systemctl restart titanit-profdnk.service`,
  - `nginx reload`.
- После деплоя workflow делает smoke-check внешнего домена:
  - `GET /api/health` должен вернуть JSON со `status=ok`,
  - `GET /login` должен вернуть HTTP `200`.

## Почтовые уведомления (SMTP)
- При создании психолога админом отправляется письмо с доступом к кабинету.
- После прохождения теста клиентом (если указан email и разрешён клиентский отчёт) отправляется ссылка на HTML-отчёт.
- SMTP настраивается только через переменные окружения / GitHub Secrets, в репозиторий секреты не коммитятся.

### Чеклист проверки SMTP в проде (VPS)
1. В GitHub Secrets заданы `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM`.
2. В `.env.deploy` на сервере есть:
   - `SMTP_ENABLED=true`
   - `SMTP_HOST=smtp.gmail.com`
   - `SMTP_PORT=587`
   - `SMTP_TLS=true`
3. После деплоя создать психолога из админки и проверить, что:
   - в логах нет `email send failed`;
   - письмо с доступом пришло на email психолога.
4. Пройти клиентский тест с заполненным `email` и включённым клиентским отчётом:
   - в логах нет `email send failed`;
   - клиент получил письмо со ссылкой на HTML-отчёт.

## Что добавили в последних итерациях

### Именованные ссылки и группировка по кампаниям
- У теста может быть несколько именованных ссылок.
- Для именованных ссылок можно задать лимит прохождений (или оставить безлимитный режим).
- После достижения лимита ссылка автоматически деактивируется.
- Прохождения группируются по метке источника.
- В деталях теста добавлены:
  - отдельная таблица групп,
  - отдельная таблица «Сравнение кампаний» со средними значениями `% балла` и `% заполнения`,
  - колонка источника и статуса ссылки в результатах,
  - фильтр результатов по статусу ссылки (активна / лимит исчерпан / отключена),
  - экспорт всех прохождений в CSV (`/tests/{id}/submissions.csv`).

Проверка:
1. Создать 2 именованные ссылки у теста (одну с лимитом `1`, вторую без лимита).
2. Пройти тест по ссылке с лимитом два раза.
3. Убедиться, что после первого прохождения ссылка помечена как «Лимит исчерпан» и второй раз недоступна.
4. Убедиться, что в результатах и группировке видны корректные метки и статусы.
5. Нажать `Экспорт CSV` и убедиться, что в файле есть `source_label`, `score_percent`, `completion_percent`.

### Динамические поля клиента + шаблоны полей
- В конструктор методик добавлен блок "Дополнительные поля клиента".
- Психолог может добавлять произвольные поля анкеты (тип, обязательность, плейсхолдер).
- Добавлены быстрые шаблоны наборов полей (например, "Школа/вуз", "Карьерная сессия").
- Поля рендерятся в клиентской форме, валидируются на сервере и попадают в отчеты.
- Для JSON-импорта поддержаны `custom_client_fields` и `client_fields.custom_fields`.

Проверка:
1. Открыть `/tests/new`, добавить 1-2 доп. поля и создать тест.
2. Пройти тест по публичной ссылке и заполнить эти поля.
3. Открыть клиентский и профессиональный HTML/DOCX-отчеты, убедиться что данные присутствуют.

### Конструктор шаблонов отчетов (блоки + пресеты)
- В конструктор методик добавлен блочный редактор структуры отчетов.
- Для клиентского и профессионального отчета настраиваются:
  - состав блоков,
  - порядок блоков,
  - быстрый пресет (базовый/демо).
- Настройка влияет на оба формата: HTML и DOCX.
- Для JSON-импорта поддержан объект `report_templates`.

Проверка:
1. Открыть `/tests/new`, в блоке шаблонов отчетов изменить порядок/состав.
2. Создать тест, пройти его и открыть HTML/DOCX отчеты.
3. Убедиться, что в отчете только выбранные блоки и в выбранном порядке.

### Быстрые пресеты методик и клонирование теста
- В ручном конструкторе доступны 3 пресета:
  - `Школьная траектория`
  - `Колледж/вуз`
  - `Карьерный перезапуск`
- Каждый пресет заполняет:
  - название и описание,
  - обязательные/дополнительные поля клиента,
  - секции и вопросы,
  - формулы метрик,
  - шаблоны клиентского и профессионального отчета.
- На странице тестов и в деталях теста доступна кнопка `Клонировать тест`.
- Клон переносит структуру методики и формулы, но не копирует кампанийные ссылки.

Проверка:
1. Открыть `/tests/new`, применить каждый пресет и убедиться, что форма заполняется автоматически.
2. Создать тест из пресета и проверить прохождение по ссылке.
3. В `/tests` нажать `Клонировать тест`.
4. Убедиться, что открылась новая карточка теста с суффиксом `(копия)`, с теми же секциями/вопросами/формулами.
