# Архитектура MVP

## Контуры
- Web UI и API реализованы в одном FastAPI-сервисе.
- Данные хранятся в PostgreSQL через SQLAlchemy ORM.
- HTML и DOCX отчёты генерируются на лету в памяти.

## Модули
- `routers/auth.py`: логин/логаут
- `routers/admin.py`: управление психологами
- `routers/psychologist.py`: кабинет, тесты, отчёты
- `routers/public.py`: публичная визитка, клиентское прохождение
- `routers/api.py`: health endpoint
- `services/tests.py`: сборка теста из UI/JSON, экспорт
- `services/scoring.py`: расчёт базовых и производных метрик
- `services/formulas.py`: безопасный вычислитель формул (AST sandbox)
- `services/reports.py`: HTML/DOCX отчёты
- `services/seed.py`: первичное наполнение БД

## Основные сущности
- `users`: админы и психологи, доступ/блокировки
- `tests`: методики и публичный токен ссылки
- `test_sections`: секции теста
- `questions`: вопросы и параметры типов
- `submissions`: прохождения клиентов
- `answers`: ответы клиента по каждому вопросу

## Поток данных
1. Психолог создаёт тест (ручной UI или JSON import).
2. Для теста создаётся `share_token`.
3. Клиент по `/t/{token}` отправляет ответы.
4. Сервис считает метрики, пишет `submission + answers`.
5. Психолог открывает отчёты:
   - HTML через шаблон рендера
   - DOCX через `python-docx` в `BytesIO`

## Безопасность MVP
- Сессионная авторизация через `SessionMiddleware`.
- Разделение ролей по server-side проверкам.
- Проверка прав на просмотр тестов/отчётов.
- Ограничение upload форматов фото.

## Ограничения MVP
- Нет Alembic-миграций (таблицы создаются на старте).
- Нет полноценного формульного конструктора (весовые метрики + score у опций).
- Пока нет branching logic и drag-and-drop.

## План архитектурного роста
- Выделить frontend на Next.js (BFF или прямой REST).
- Ввести Alembic и версионирование схемы.
- Добавить background tasks и кэш для тяжёлых отчётов.
- Внедрить объектное хранилище для медиа и CDN.
